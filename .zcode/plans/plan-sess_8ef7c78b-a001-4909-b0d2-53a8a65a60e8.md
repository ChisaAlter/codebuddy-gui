# 计划：store 渐进拆分 + 关键路径集成测试 + CLI L1 兼容管理

## 目标

1. **渐进拆 `store.js`**：按领域物理分文件，**行为与对外 API 不变**（仍 `export const useStore` / 既有 helper 导出名）。
2. **补关键路径薄集成测试**：新会话首条消息、权限卡去重、退出同步保存。
3. **设置里加强 CodeBuddy CLI 管理（L1）**：**不内置 CLI**；声明兼容版本，**低于 min 硬阻断 `runtime.ensure`**，设置页引导一键安装推荐版。

---

## 为何不做「内置 CLI」

当前所有 spawn 都走 PATH 上的裸命令 `codebuddy`（`codebuddy-runtime-manager.cjs`、`main.cjs` 的 `runCodeBuddyCli`），安装包 `files` 不含 CLI。内置（L3）会带来：

- 安装包体积暴涨、签名/供应链、与终端全局 CLI 双份
- GUI 与 CLI 发布节奏绑死

设置里已有 doctor / update / install 指定版本。**L1 兼容锁 + 硬阻断**成本最低、收益最大。

---

## A. store.js 渐进物理拆分

### 原则

- **只搬家，不改状态形状**：保留 active-thread 字段双写（`ACTIVE_THREAD_RUNTIME_KEYS` / `patchThreadRuntime`）。
- **`src/store.js` 继续是唯一公共入口**：`export const useStore = create(...)`，`export { conversations }`，以及 `hasCompletePromptResponse`、`runtimeAuthScopeChanged`。
- **组件与现有单测 import 路径不变**（`from '../store'` / `../../src/store`）。
- 新增目录：`src/store/`（slices + helpers），避免与 `src/store.js` 冲突时可用：
  - `src/store.js`（组装）
  - `src/store/helpers/*.js`、`src/store/slices/*.js`

### 建议文件切分

| 文件 | 内容（对照现有区间） |
|------|----------------------|
| `src/store/helpers/prompt-completion.js` | `hasCompletePromptResponse`、busy statuses |
| `src/store/helpers/runtime-auth.js` | `runtimeAuthScopeChanged` |
| `src/store/helpers/queues.js` | 导航/runtime/thread/prompt 队列、request scope |
| `src/store/helpers/thread-runtime.js` | `emptyThreadRuntime`、`ACTIVE_THREAD_RUNTIME_KEYS`、mirror helpers |
| `src/store/helpers/terminal-workspace-state.js` | `makePane`、terminal/workspace 快照 helper |
| `src/store/module-state.js` | `conversations` 单例、bootstrap flags、persist timer Maps（模块级共享） |
| `src/store/slices/product-persist.js` | `persistProductState`、`flushProductStateSync`、hydrate 相关 action 工厂 |
| `src/store/slices/projects-runtime.js` | 项目 CRUD、`ensure/start/stop/restart`、`applyProjectRuntimeStatus` |
| `src/store/slices/sessions-chat.js` | thread 生命周期、`handleConversationEvent`、`sendPrompt`/`runThreadPrompt`、权限/问题 |
| `src/store/slices/workspace-files.js` | 文件树/预览/保存/watcher/dirty confirm |
| `src/store/slices/terminal.js` | terminal panes / PTY bind |
| `src/store/slices/ops-views.js` | workers/plugins/stats/tasks/traces/settings refresh |
| `src/store/slices/bootstrap-auth.js` | `bootstrap`、login/logout/refresh |
| `src/store.js` | `create((set,get) => ({ ...initial, ...sliceA(set,get), ... }))` |

Slice 形态统一为：

```js
export function createXxxSlice(set, get, ctx) {
  return {
    async ensureProjectRuntime(...) { ... },
    // ...
  };
}
```

`ctx` 注入模块级依赖（`conversations`、timer maps、queue helpers），避免 slice 互相 `import store` 造成循环。

### 落地顺序（每步保持 `npm test` 绿）

1. 抽出 **纯 helper**（`hasCompletePromptResponse` 等）+ 让 `store.js` re-export → 现有测试零改或只改 import 到 store 再导出。
2. 抽出 **module-state + queues + thread-runtime helpers**。
3. 按依赖从叶到根迁 action：`ops-views` → `workspace-files` → `terminal` → `product-persist` → `projects-runtime` → `sessions-chat` → `bootstrap-auth`。
4. 最终 `store.js` 只剩 create 组装 + re-export（目标：主文件 &lt; ~300 行）。

### 明确不做（本阶段）

- 不改 Zustand 多 store / 不拆 `useStore` 实例
- 不消减 active-thread 双写
- 不强制改组件 selector（可顺手改明显整包订阅，但非必须）

### 验收

- 全量 `npm test` 316+ 仍通过
- `git diff` 行为面：逻辑行尽量「移动」为主
- 开发模式 smoke：启动 → 开项目 → 发消息 → 切对话 → 退出保存

## B. 关键路径薄集成测试

优先 **store 集成**（mock `getThreadClient` / `window.electronAPI`），不把竞态塞进全量 Electron e2e。

### B1 扩展 `tests/unit/store-prompt-session.test.js`

- **双 null**：thread.sessionId 与 runtime.sessionId 皆空 → 不调 `session/prompt`，错误「尚未完成连接」
- **thread 记录滞后、runtime 已有 id**（已有则保留作锚）
- **飞行中 session 切换**：`requestSessionId` 与后续 patch 不一致时不写脏成功态
- `hasCompletePromptResponse` 负例（仅 thinking/tools、空 assistant）

### B2 新建 `tests/unit/store-permission-dedupe.test.js`

- 同 `toolCallId` / 同 `interruptionId` 双通道合并为一条
- 重复 payload 不叠加
- 快速双击 `respondToInterruption('allow')` → 客户端只成功一次
- `interaction_requests_invalidated` 清空 + respond 失败
- 参考：`store-conversation-events` + `store-cancel` 的 mock 风格

### B3 新建 `tests/unit/store-product-state-flush.test.js`

- 挂起 timeline debounce 后 `flushProductStateSync` 写入最新 timeline
- 无 `saveProductStateSync` → false
- `{ ok: false }` → 置错误、返回 false
- 并发 `persistProductState` 后 flush 不抛、快照最终一致

### B4 新建 `tests/unit/product-state-store.test.js`（Node，require `electron/product-state.cjs`）

- save/load 往返、主文件损坏回 `.bak`、原子写、非法 JSON 隔离

### 不做

- 不为这些竞态扩 e2e（易 flaky）；e2e 维持「新会话 id 可见」烟雾即可。

## C. CLI L1：兼容锁 + 设置增强 + 硬阻断

### 产品行为

1. 在主进程集中声明兼容矩阵，例如：

```js
// electron/cli-compat.cjs（新建）
module.exports = {
  minVersion: '2.122.0',
  recommendedVersion: '2.122.0',
  // 可选：maxVersion 暂不硬拦「过高」
};
```

2. `getCodeBuddyCliInfo()` 扩展返回：

- `version` / `output` / `truncated`（现有）
- `compat`: `{ status: 'ok'|'outdated'|'missing'|'unknown', minVersion, recommendedVersion, message }`
- 用已有 `compareVersions` 做比较

3. **硬阻断启动运行时**  
   在 `runtime:ensure`（`main.cjs` handler → `runtimeManager.ensure` 前）或 `ensure` 入口：

- 探测版本；`missing` / `outdated`（&lt; min）→ **抛明确错误**，不 spawn `--serve`
- 错误文案含推荐版本与「打开设置 → CLI 维护安装」指引
- 渲染层已有：`ensureProjectRuntime` catch → `runtimeStatus: 'error'` + `connectionState: 'error'`（`store.js` ~2756–2762）

4. **设置页增强**（`ReplicaSettingsView.jsx` CLI 维护区块）

- 展示：当前版本、**最低/推荐版本**、兼容状态徽章（可用 / 过低 / 缺失）
- 过低/缺失时突出 **「安装推荐版本 vX.Y.Z」** 一键（复用 `installCodeBuddyCli(recommended)`）
- `update` 按钮：若当前已 ≥ recommended，文案可提示「更新可能超出 GUI 验证范围」（不阻断，或二次确认——默认 **警告可继续**）
- 安装/更新成功后仍保留「重启当前运行时」

5. **安装缺 CLI 的鸡生蛋**  
   今日 `installCodeBuddyCli` 会先 `getCodeBuddyCliInfo()`（CLI 不存在则失败）。L1 内：

- `before` 探测失败时：`beforeVersion: null`，仍尝试 `codebuddy install <target>`（若 PATH 上已有 npm 全局 shim / 安装器入口）
- 若 install 命令本身也 ENOENT：明确错误「未找到 codebuddy，请先按官方文档安装 CLI，再回设置安装推荐版本」+ 可选打开文档链接  
  （L1 **不**做离线包下载）

6. **文档**  
   `README.md` / `CODEBUDDY.md` / `RELEASE_NOTES.md` 写明：本 GUI 支持的 min/recommended CLI；不兼容时拒绝启动项目运行时。

### 测试（CLI）

- 单元：`compareVersions` + compat 判定（`tests/unit/cli-compat.test.js`）
- store/bootstrap 可选：`ensureProjectRuntime` 在 IPC 抛 compat 错误时写入 error 状态（mock `electronAPI`）

### 明确不做（本阶段）

- 不打包 CLI 进安装包（L3）
- 不写 userData 托管安装（L2）
- 不改 PATH 发现模型、不加自定义 CLI 路径（你选的是纯 L1；路径覆盖可作后续 L1.5）

```text
Phase 0  测试加固（B1–B4）——给拆分与 CLI 改动垫安全网
Phase 1  store helper 抽出 + re-export（A 步 1–2）
Phase 2  store slices 迁移动作（A 步 3–4），每步 npm test
```

## 风险与缓解

| 风险 | 缓解 |
|------|------|
| 大文件搬家合并冲突 | 小步 PR/commit：helpers → 单 slice → 下一 slice |
| slice 循环依赖 | 模块状态与 queues 放 `module-state`/`helpers`，slice 只收 `set/get/ctx` |
| 硬阻断误伤新 CLI | 仅 `version < min` 阻断；高于 recommended 只警告 |
| install 仍依赖系统已有 CLI 入口 | L1 接受；文案引导官方安装；L2 再做托管下载 |
| flush/exit 假定时器 flaky | 测 flush 直接清 timer 路径，避免真实 sleep |

## 手测清单（完成后）

1. CLI &lt; min：添加项目 / 启动运行时失败，错误可读；设置页一键装推荐版 → 重启运行时 → 可聊天  
2. CLI = recommended：正常多项目、发消息、权限卡、托盘完全退出状态恢复

4. 退出时有未落盘草稿/timeline：`flushProductStateSync` 后重启仍在
